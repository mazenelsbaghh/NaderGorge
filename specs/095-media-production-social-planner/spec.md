# Feature Specification: Media Production Pipeline and Social Planner

**Feature Branch**: `095-media-production-social-planner`  
**Created**: 2026-06-09  
**Status**: Draft  
**Input**: User description: "Phase 8 - Media Production Pipeline and Social Planner"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Media Production Pipeline Tracking (Priority: P1)

As a content manager or supervisor, I want to track media production items (such as lessons, promotional videos, or announcements) through a standardized pipeline from preparation to final publication, so that I can monitor progress, assign tasks, and track performance.

**Why this priority**: It is the core operational database and tracking system for the media team, establishing the standard production stages, assignee attributes, and metadata.

**Independent Test**: Can create a pipeline item, assign an editor/producer, add asset URLs, increment the editing error count, and transition it through stages up to `Review` independently of the approval or scheduling systems.

**Acceptance Scenarios**:

1. **Given** a content manager is logged in, **When** they create a new pipeline item with a title, description, target publication date, and assigned editor, **Then** the item is successfully saved with the stage set to `Preparation` and `PublishedAt` set to null.
2. **Given** a pipeline item is in `Editing` stage, **When** the editor updates the asset folder URL or increments the editing error count, **Then** the updates are saved and visible.
3. **Given** a pipeline item is in `Review` stage, **When** a non-admin user attempts to transition it directly to `Published` without approval, **Then** the request is blocked.
4. **Given** a pipeline item is in `Approved` stage, **When** the manager transitions it to `Published`, **Then** the stage changes to `Published` and `PublishedAt` is set to the current UTC timestamp.

---

### User Story 2 - Integrated Content Approval Workflow (Priority: P2)

As a media editor or creator, I want my finished content to be reviewed and approved by a supervisor before it is published, so that we ensure high academic and visual quality.

**Why this priority**: Connects the production pipeline to the platform's approval system, enforcing quality control and preventing unauthorized publishing of content.

**Independent Test**: Transitioning a pipeline item to the `Review` stage automatically creates an approval request in the system, and resolving that request updates the pipeline item.

**Acceptance Scenarios**:

1. **Given** a pipeline item is in `Editing` or `Uploading` stage, **When** the assignee transitions it to `Review`, **Then** the system automatically generates a pending approval request linked to this pipeline item.
2. **Given** a pending approval request for a pipeline item, **When** a supervisor approves the request, **Then** the approval request is marked as completed, and the pipeline item's stage is automatically updated to `Approved`.
3. **Given** a pending approval request for a pipeline item, **When** a supervisor rejects the request with notes, **Then** the approval request is marked as rejected, and the pipeline item's stage is automatically reverted to `Editing` with the rejection notes appended to its history.

---

### User Story 3 - Social Media Planner and Scheduling (Priority: P3)

As a social media manager, I want to plan and schedule social media posts (including content ideas, scripts, and publication channels), so that we maintain a consistent marketing presence.

**Why this priority**: Helps coordinate social content creation with the primary media production, allowing scripting and ideas to be structured in a visual planner.

**Independent Test**: Can create and manage social planner entries, set publication channels, write scripts, and link them optional to media production pipeline items.

**Acceptance Scenarios**:

1. **Given** a social media manager is on the planner page, **When** they create a new social plan with an idea title, script draft, target platform (e.g., YouTube, TikTok, Facebook), and scheduled date, **Then** the plan is saved in `Scripting` status.
2. **Given** a social plan exists, **When** the manager links it to a media production pipeline item, **Then** the linkage is saved and visible on both entities.
3. **Given** a social plan is linked to a pipeline item, **When** the pipeline item's stage is updated, **Then** the social planner displays the updated status of the linked media asset.

---

### User Story 4 - KPIs and Production Analytics (Priority: P4)

As a supervisor or manager, I want to view performance metrics and KPIs for the media production team, so that I can identify bottlenecks and track editor quality.

**Why this priority**: Essential for business visibility and tracking the performance of the production team, but relies on logs generated by the other stories.

**Independent Test**: Can query aggregated stats (total assets produced, average days per stage, and editor error counts) for a given date range.

**Acceptance Scenarios**:

1. **Given** multiple pipeline items have been processed and published, **When** a manager loads the media KPIs dashboard, **Then** the dashboard displays the total assets published, average time spent in the `Editing` stage, and a list of editors sorted by their total editing error counts.
2. **Given** a date range filter is applied, **When** the supervisor requests reports, **Then** the statistics are filtered accurately to items published or worked on within that range.

### Edge Cases

- **Duplicate Asset URLs**: If multiple pipeline items reference the same Google Drive/OneDrive folder link, the system should allow it but display a subtle warning to the creator to prevent accidental work duplication.
- **Assignee Account Deactivation**: If an editor or producer user account is deactivated while they have active production items, the system must retain the pipeline items, set the assignee to null, and flag the items as "Unassigned" in the dashboard.
- **Rejection Loop**: If a pipeline item is rejected, corrected, and resubmitted multiple times, the system must keep a chronological log of all approval attempts, outcomes, and comments so the history is transparent.

### Manual QA & Docker Acceptance *(mandatory)*

- **Manual QA Manager Flow**: Admin log in, access `/admin/media`, create a pipeline item, assign it to an assistant user, and transition it to `Filming` then `Editing`.
- **Manual QA Editor Flow**: Assistant log in, access `/assistant/media` (or operations dashboard), view the assigned item, update the asset folder URL, increment error count by 2, and submit for `Review`.
- **Manual QA Supervisor Flow**: Supervisor log in, access approvals panel, view the new media review item, reject it first (assert it goes back to `Editing`), then after editor resubmits, approve it (assert it goes to `Approved`). Manager then publishes it (assert `PublishedAt` is set).
- **Manual QA Negative Check**: A student user trying to access `/api/admin/media` or `/admin/media` must receive a 403 Forbidden/401 Unauthorized block. An editor (Assistant) trying to approve their own media item or change stage directly to `Approved`/`Published` must be blocked.
- **Docker Acceptance**: 
  - `make up` launches all services (including backend, frontend, postgres, and redis).
  - `make migrate` runs the database migrations containing the media production tables.
  - Health check at `/api/health` returns healthy.
- **External Dependencies**: None. The social media planner is a coordination tool and does not connect to external APIs (Facebook Graph API, YouTube Data API, etc.) for direct auto-posting in this phase.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The system MUST support a `MediaProductionPipeline` entity containing fields for title, description, stage (Preparation, Filming, Editing, Uploading, Review, Approved, Published), assigned agent (User), asset folder URL (string), editing error count (integer), and publication timestamp (`PublishedAt`).
- **FR-002**: Only users with the `media.manage` permission claim or `Admin`/`Supervisor` roles MUST be allowed to create, update, or assign media pipeline items.
- **FR-003**: The system MUST automatically update the `PublishedAt` timestamp of a pipeline item to the current UTC time when its stage transitions to `Published`, and clear it if it transitions back.
- **FR-004**: The system MUST prevent direct transition of a pipeline item from `Review` (or earlier stages) to `Published` without first transitioning to `Approved` via the approval workflow.
- **FR-005**: The system MUST support a `SocialMediaPlan` entity containing title, description, script/copy, target platform (YouTube, Facebook, Instagram, TikTok, Telegram), scheduled publication date, status (Draft, Scripting, Scheduled, Published), and an optional foreign key link to a `MediaProductionPipeline` item.
- **FR-006**: When a pipeline item transitions to the `Review` stage, the system MUST automatically create a corresponding `ApprovalRequest` entity (reusing the approval pipeline database model).
- **FR-007**: The system MUST expose API endpoints for listing pipeline items with support for search, status filtering, assignee filtering, and pagination.
- **FR-008**: The system MUST expose API endpoints for media metrics, returning total assets produced, average duration in editing stage, and error counts grouped by editor.

### Key Entities *(include if feature involves data)*

- **MediaProductionPipeline**: Represents a video or media asset in progress.
  - `Id` (Guid, PK)
  - `Title` (string, required)
  - `Description` (string, optional)
  - `Stage` (enum: Preparation = 0, Filming = 1, Editing = 2, Uploading = 3, Review = 4, Approved = 5, Published = 6)
  - `AssignedAgentId` (Guid, Nullable FK to Users)
  - `AssetFolderUrl` (string, optional)
  - `EditingErrorCount` (int, default 0)
  - `PublishedAt` (DateTime, Nullable)
  - `CreatedAt` (DateTime)
  - `UpdatedAt` (DateTime)
- **SocialMediaPlan**: Represents a social marketing schedule entry.
  - `Id` (Guid, PK)
  - `Title` (string, required)
  - `Description` (string, optional)
  - `Script` (string, optional)
  - `Platform` (enum: YouTube = 0, Facebook = 1, Instagram = 2, TikTok = 3, Telegram = 4)
  - `Status` (enum: Draft = 0, Scripting = 1, Scheduled = 2, Published = 3)
  - `ScheduledDate` (DateTime, required)
  - `MediaProductionPipelineId` (Guid, Nullable FK to MediaProductionPipeline)
  - `CreatedAt` (DateTime)

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A content manager can create a pipeline item and assign an editor in under 15 seconds.
- **SC-002**: Transitioning an item to `Review` creates the corresponding approval request in the database within 500ms.
- **SC-003**: The media dashboard displays updated performance metrics (time in stages, error rates) aggregated from production records in under 2 seconds.
- **SC-004**: 100% of published media pipeline assets must have a recorded approval signature from a supervisor.

## Assumptions

- **Manual Posting**: Social media posts scheduled in the planner will be posted manually by the marketing team. No automated integrations with third-party social media APIs are in scope.
- **Asset Storage**: Asset folders are hosted on external drives (like Google Drive). The system only stores URLs to these folders, not the actual large video files.
- **Existing Approval Infrastructure**: We will reuse the existing approval system tables and endpoints for pipeline reviews, mapping a new source type for "MediaProduction".
