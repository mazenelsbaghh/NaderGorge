# Feature Specification: SMS Payment Auto-Matcher

**Feature Branch**: `148-sms-payment-auto-matcher`  
**Created**: 2026-06-25  
**Status**: Draft  
**Input**: User description: "Build an Android app that listens to incoming SMS messages from digital wallets (e.g., Vodafone Cash), sends them to the server for automatic matching with student recharge requests, and provides a manual verification screen for unmatched transactions. The app supports multiple paired phones with daily/monthly limits, shows all connected devices and their balances, and uses a pairing code (no login) for setup."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Admin Creates a Digital Wallet & Generates Pairing Code (Priority: P1)

An admin navigates to the "Digital Wallets" section in the web admin dashboard. They create a new wallet by entering the phone number (e.g., "01012345678"), a label (e.g., "Vodafone Cash 1"), the daily receiving limit (e.g., 30,000 EGP), and the monthly receiving limit (e.g., 100,000 EGP). Upon creation, the system generates a unique pairing code. The admin copies this code to configure the Android device.

**Why this priority**: Without wallets and pairing codes, no Android device can connect and no recharge flow can function. This is the foundational setup step.

**Independent Test**: Can be tested by creating a wallet in the admin panel and verifying the pairing code is generated and displayed.

**Acceptance Scenarios**:

1. **Given** the admin is on the wallet management page, **When** they fill in all required fields and submit, **Then** a new wallet record is created with status "Disconnected" and a unique 8-character pairing code is generated and displayed.
2. **Given** a wallet already exists with phone number "01012345678", **When** the admin tries to create another wallet with the same phone number, **Then** the system rejects it with a duplicate error.
3. **Given** a wallet exists, **When** the admin edits its daily or monthly limit, **Then** the new limits take effect immediately for subsequent recharge requests.

---

### User Story 2 - Android App Pairs with Server Using Pairing Code (Priority: P1)

A staff member installs the Android app on a phone that has a Vodafone Cash SIM. Upon first launch, the app shows a setup screen with two fields: Server URL and Pairing Code. The staff member enters the server URL and the pairing code generated by the admin. The app validates the code with the server. On success, the app saves the pairing token locally, transitions to the dashboard screen, and begins listening for SMS messages in the background.

**Why this priority**: This is the core connection mechanism. Without pairing, the Android app cannot communicate with the server.

**Independent Test**: Can be tested by entering a valid pairing code and confirming the app transitions to the dashboard showing wallet info.

**Acceptance Scenarios**:

1. **Given** the app is freshly installed, **When** the user enters a valid server URL and pairing code, **Then** the app verifies the code with the server, saves the token, and shows the dashboard with the wallet's phone number and current balance.
2. **Given** the app is freshly installed, **When** the user enters an invalid pairing code, **Then** the app displays an error message "كود الربط غير صحيح" (Invalid pairing code) and stays on the setup screen.
3. **Given** the app is already paired, **When** the app is reopened, **Then** it skips the setup screen and goes directly to the dashboard.

---

### User Story 3 - Android App Captures and Forwards SMS (Priority: P1)

The Android app runs a background SMS receiver service. When an SMS arrives from a sender matching the configured filter names (e.g., "VodafoneCash", "VF-Cash"), the app immediately forwards the message body, sender name, and receipt timestamp to the server via the API. If the internet is unavailable, the message is queued locally and sent when connectivity is restored.

**Why this priority**: This is the core data-capture mechanism. Without SMS forwarding, automatic matching cannot happen.

**Independent Test**: Can be tested by simulating an SMS from "VodafoneCash" on an emulator and verifying the message appears in the server's SMS log.

**Acceptance Scenarios**:

1. **Given** the app is paired and running in the background, **When** an SMS arrives from "VodafoneCash", **Then** the app extracts the sender, body, and timestamp and sends them to `POST /api/android/sms` within 5 seconds.
2. **Given** the app is paired and has no internet, **When** an SMS arrives from a matching sender, **Then** the message is stored locally and automatically synced when connectivity is restored (within 30 seconds of reconnection).
3. **Given** the app is paired, **When** an SMS arrives from a non-matching sender (e.g., a personal contact), **Then** the app ignores the message entirely and does not send it to the server.

---

### User Story 4 - Student Initiates Recharge Request (Priority: P1)

A student navigates to the recharge page on the website. They enter the amount they want to top up (e.g., 100 EGP). The system selects an available wallet that has not exceeded its daily or monthly limits and displays the wallet's phone number to the student. The student is given 20 minutes to complete the transfer. The student then enters the phone number they transferred from, the actual amount they transferred, uploads a screenshot of the transfer confirmation, and submits the request.

**Why this priority**: This is the student-facing entry point for the entire payment flow. Without it, students cannot initiate recharges.

**Independent Test**: Can be tested by initiating a recharge, verifying a wallet number is displayed, and confirming the request is saved as "Pending."

**Acceptance Scenarios**:

1. **Given** there are active wallets with remaining daily capacity, **When** a student initiates a recharge for 100 EGP, **Then** the system assigns an available wallet and displays its phone number with a 20-minute countdown timer.
2. **Given** all wallets have reached their daily limits, **When** a student tries to initiate a recharge, **Then** the system displays a message "عذراً، خدمة الشحن غير متاحة حالياً. حاول مرة أخرى لاحقاً." (Sorry, recharge service is currently unavailable. Try again later.)
3. **Given** a wallet has been assigned to a student, **When** the student submits the form with sender phone number, transferred amount, and screenshot, **Then** a RechargeRequest is created with status "Pending" and linked to the assigned wallet.
4. **Given** the 20-minute reservation has expired without submission, **When** checked, **Then** the reservation is released and the wallet capacity is restored for other students.

---

### User Story 5 - Server Auto-Matches SMS with Recharge Request (Priority: P1)

When the server receives an SMS from the Android app, it parses the message body using regular expressions to extract the transfer amount and sender phone number. It then searches for a pending RechargeRequest with matching amount and sender phone number created within the last 2 hours on the same wallet. If a match is found, the student's balance is credited immediately, the request status changes to "Matched", and the wallet's tracked balance is updated.

**Why this priority**: This is the core automation that eliminates manual work. Without auto-matching, the system adds overhead rather than reducing it.

**Independent Test**: Can be tested by creating a pending request and simulating an SMS that matches, then verifying balance is credited.

**Acceptance Scenarios**:

1. **Given** a student created a recharge request for 100 EGP from phone "01098765432" on wallet "01012345678", **When** the server receives an SMS on that wallet confirming receipt of 100 EGP from "01098765432", **Then** the student's balance is credited with 100 EGP, the request status becomes "Matched", and the wallet balance is updated.
2. **Given** no pending request matches the incoming SMS, **When** the server processes the SMS, **Then** the SMS is logged as "Unmatched" and an alert is sent to the admin verification queue.
3. **Given** the same SMS is received twice (duplicate), **When** the server processes the second copy, **Then** it recognizes the duplicate and skips processing without creating a double credit.

---

### User Story 6 - Admin Reviews Unmatched Transactions (Priority: P2)

An admin navigates to the "Pending Verification" screen in the web dashboard. They see a list of unmatched recharge requests alongside unmatched SMS messages. Each entry shows the student's name, requested amount, sender phone, screenshot, and any related SMS logs. The admin can approve (manually matching a request to an SMS and crediting the student) or reject (marking the request as rejected with a reason).

**Why this priority**: Manual review is the safety net. Without it, legitimate payments that fail auto-matching would be stuck forever.

**Independent Test**: Can be tested by creating an unmatched request, navigating to the verification queue, and approving it manually.

**Acceptance Scenarios**:

1. **Given** there are 3 unmatched recharge requests, **When** the admin opens the verification screen, **Then** all 3 requests are displayed with student info, amount, sender phone, screenshot preview, and timestamp.
2. **Given** an unmatched request, **When** the admin clicks "Approve" and confirms, **Then** the student's balance is credited, the request status changes to "Approved", and the entry disappears from the queue.
3. **Given** an unmatched request, **When** the admin clicks "Reject" and enters a reason, **Then** the request status changes to "Rejected", the entry disappears from the queue, and the rejection reason is stored.

---

### User Story 7 - Android App Shows All Connected Devices Dashboard (Priority: P2)

The Android app dashboard displays the current device's wallet information (phone number, current balance, daily limit, monthly limit, amount received today, amount received this month) and a list of all other connected devices in the system with their wallet info and connection status (online/offline).

**Why this priority**: Provides operational visibility to staff managing multiple phones, but is not strictly required for the core recharge flow.

**Independent Test**: Can be tested by pairing two devices and verifying each device shows the other's wallet info.

**Acceptance Scenarios**:

1. **Given** 3 devices are paired with the system, **When** device A opens the dashboard, **Then** it shows its own wallet details and the wallet details of devices B and C with their connection status.
2. **Given** device B goes offline, **When** device A refreshes the dashboard, **Then** device B is shown with status "غير متصل" (Disconnected).
3. **Given** a new SMS is received on device B, **When** device A refreshes, **Then** device B's balance and "received today" amount are updated.

---

### User Story 8 - Admin Manages Wallet Limits and Status (Priority: P2)

An admin can edit wallet settings (daily limit, monthly limit, label), activate/deactivate wallets, regenerate pairing codes (which disconnects the current device), and view transaction history per wallet.

**Why this priority**: Operational management is needed for day-to-day operations but can be done with direct database access in an emergency.

**Independent Test**: Can be tested by editing a wallet's limit and verifying the new limit is enforced on the next recharge request.

**Acceptance Scenarios**:

1. **Given** a wallet with daily limit 30,000 EGP, **When** the admin changes it to 50,000 EGP, **Then** the new limit takes effect immediately.
2. **Given** an active wallet, **When** the admin deactivates it, **Then** the wallet is no longer offered to students for recharge and shows as "غير نشط" (Inactive) in the dashboard.
3. **Given** a paired device, **When** the admin regenerates the pairing code, **Then** the old device is disconnected, the new code is displayed, and the old device must re-pair.

### Edge Cases

- What happens when a student transfers a different amount than what they entered in the form? The SMS amount will not match the request amount, causing it to go to manual review.
- What happens when two students transfer the same amount from the same phone number around the same time? The system matches the oldest pending request first (FIFO). The second transfer goes to manual review.
- What happens when the Android app is force-killed by the OS? The SMS BroadcastReceiver is registered in the manifest and will be triggered by the OS even if the app is not running. WorkManager ensures the SMS is delivered to the server.
- What happens when the server is temporarily unreachable? The Android app queues messages locally using WorkManager with exponential backoff retry until successful delivery.
- What happens when a wallet's daily limit is reached mid-day but there are pending reservations? Existing reservations are honored. New students see other available wallets or the "service unavailable" message.

### Manual QA & Docker Acceptance *(mandatory)*

- **Manual QA Admin Flow**: Admin logs in → navigates to Wallets → creates wallet with limits → copies pairing code → verifies wallet appears in list with "Disconnected" status.
- **Manual QA Android Pairing**: Install APK on device → enter server URL and pairing code → verify dashboard loads with wallet info → verify device status changes to "Connected" on admin panel.
- **Manual QA Student Recharge**: Student logs in → navigates to Recharge → enters amount → sees wallet phone number → submits with sender phone, amount, and screenshot → verifies request is "Pending".
- **Manual QA Auto-Match**: Send test SMS to paired device → verify student balance is credited → verify request status is "Matched".
- **Manual QA Manual Review**: Create an unmatched request → admin opens verification queue → approves → verify student balance is credited.
- **Manual QA Negative Check**: Student with no available wallets cannot initiate recharge. Invalid pairing code is rejected by Android app.
- **Docker Acceptance**: Backend compiles and runs. Database migration creates new tables (DigitalWallet, RechargeRequest, IncomingSmsLog). API endpoints respond correctly. Frontend pages load without errors.
- **External Dependencies**: An Android device or emulator with SMS capability for full end-to-end testing.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST allow admins to create, edit, activate, and deactivate digital wallets with phone number, label, daily limit, and monthly limit.
- **FR-002**: System MUST generate a unique 8-character alphanumeric pairing code for each wallet upon creation.
- **FR-003**: Android app MUST authenticate with the server using only the pairing code (no username/password login).
- **FR-004**: Android app MUST listen for incoming SMS messages from sender names configured via the web admin panel (synced to the device via the sync-status API) in the background, even when the app is not in the foreground. The sender filter list MUST be updatable without modifying or reinstalling the Android app.
- **FR-005**: Android app MUST forward captured SMS messages (sender, body, timestamp) to the server immediately, or queue them locally if offline and retry automatically when connectivity is restored.
- **FR-006**: Server MUST parse incoming SMS bodies using regular expressions to extract transfer amount and sender phone number.
- **FR-007**: Server MUST automatically match incoming SMS data against pending recharge requests by comparing amount, sender phone number, and wallet, within a 2-hour time window.
- **FR-008**: Upon successful auto-match, the server MUST credit the student's balance immediately and update the wallet's tracked balance and daily/monthly received totals.
- **FR-009**: Unmatched SMS messages MUST be flagged and displayed in a manual verification queue for admin review.
- **FR-010**: Students MUST be able to initiate a recharge by specifying the desired amount.
- **FR-011**: System MUST select an available wallet (active, within daily and monthly limits) and display its phone number to the student with a 20-minute reservation window.
- **FR-012**: Students MUST provide the sender phone number, the actual transferred amount, and a screenshot when submitting a recharge request.
- **FR-013**: System MUST prevent duplicate SMS processing by checking message body and timestamp uniqueness per wallet.
- **FR-014**: System MUST release wallet reservations after the 20-minute timeout if the student does not submit.
- **FR-015**: Admins MUST be able to approve or reject unmatched recharge requests from the verification queue, with a mandatory rejection reason.
- **FR-016**: Android app dashboard MUST display the current device's wallet info and a list of all other connected devices with their wallet balances and connection status.
- **FR-017**: System MUST track daily and monthly received amounts per wallet and exclude wallets that have reached their limits from being offered to students.
- **FR-018**: Android app MUST send periodic heartbeat/sync requests to the server to update device connection status and receive the latest sender filter list and wallet configuration.
- **FR-019**: Admins MUST be able to add, edit, and remove SMS sender filter names (e.g., "VodafoneCash", "VF-Cash") from the web admin panel. Changes MUST be synced to all connected Android devices on the next heartbeat/sync cycle.

### Key Entities

- **DigitalWallet**: Represents a mobile money wallet (phone number, label, limits, balance, pairing token, device status). One wallet per phone number.
- **RechargeRequest**: Represents a student's recharge attempt (student, wallet, amount, sender phone, screenshot, status lifecycle: Pending → Matched/Approved/Rejected). Links a student to a specific wallet and transfer.
- **IncomingSmsLog**: Represents a captured SMS message from the Android app (wallet, sender name, message body, timestamp, match status). Used for audit trail and duplicate detection.
- **StudentBalance**: Existing entity that tracks the student's available balance. Modified by successful recharge matching.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 80% or more of legitimate student transfers are auto-matched and credited within 60 seconds of the SMS being received on the phone, without any manual intervention.
- **SC-002**: Manual review time for unmatched transactions is reduced to under 30 seconds per transaction (view screenshot, compare with SMS log, approve/reject).
- **SC-003**: Zero duplicate credits occur from the same transfer (duplicate SMS detection works correctly).
- **SC-004**: Students see the wallet phone number and can submit their recharge request in under 2 minutes from initiating the flow.
- **SC-005**: Android app successfully delivers captured SMS to the server within 10 seconds when online, and within 60 seconds of reconnecting when recovering from an offline period.
- **SC-006**: Admin can view the real-time status of all connected devices and their wallet balances from the web dashboard.

## Assumptions

- Students have stable internet to access the website and submit recharge requests.
- The Android devices used for SMS capture are dedicated to this purpose and have reliable cellular signal for SMS reception.
- Vodafone Cash and similar services send consistent SMS formats that can be reliably parsed with regular expressions.
- The existing StudentBalance and BalanceTransaction infrastructure will be reused for crediting student balances.
- The Android app targets Android 8.0 (API 26) and above, which covers the vast majority of devices in use.
- The server's base URL is accessible from the Android devices (either on the same network or via public internet).
- File uploads (screenshots) use the existing file upload infrastructure on the backend.
